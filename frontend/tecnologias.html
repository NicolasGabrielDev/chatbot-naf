Aplicação

Chatbot fiscal com RAG local

O projeto combina uma API JSON em FastAPI, serviços Python para IA e recuperação de contexto, ChromaDB persistido em disco e uma interface web estática servida pelo próprio backend.

Backend

Python e FastAPI

A API é criada em app/main.py, registra as rotas de app/api/routes.py e serve os arquivos de frontend/ com StaticFiles.

  • GET /health retorna o status da API.
  • POST /chat recebe a pergunta e retorna resposta, fontes e modelo usado.
  • Pydantic valida a entrada, saída e variáveis de configuração.
Frontend

HTML, CSS e JavaScript

A interface principal fica em frontend/index.html, frontend/app.js e frontend/styles.css, sem framework de frontend.

  • Componentes são funções JavaScript simples.
  • O chat envia mensagens para /chat via fetch.
  • A sessão usa crypto.randomUUID() quando disponível.
IA

OpenAI ou Gemini

O serviço LLMService seleciona o provedor por variável de ambiente e centraliza geração de resposta, classificação temática e embeddings externos quando configurados.

  • LLM_PROVIDER aceita openai ou gemini.
  • LLM_MODEL define o modelo de resposta.
  • Erros de cota são tratados com resposta HTTP 429.
RAG

ChromaDB e embeddings

O contexto é dividido em chunks, convertido em embeddings e salvo em uma coleção local do ChromaDB no diretório app/vectorstore.

  • O padrão atual usa embedding local por hashing.
  • TOP_K limita a quantidade de trechos recuperados.
  • A implementação atual consulta a coleção de forma vetorial global.
Fluxo

Como uma pergunta é processada

  1. A API valida a pergunta e registra um X-Request-ID.
  2. Perguntas diretas sobre escopo ou contador podem receber resposta local.
  3. Mensagens curtas ou vagas podem receber uma resposta curta do provedor configurado.
  4. Para perguntas fiscais, o RAG consulta o ChromaDB usando embedding da pergunta.
  5. Os trechos recuperados e o prompt de sistema são enviados ao modelo.
  6. A resposta final retorna em JSON com fontes e modelo utilizado.
Fonte

Base oficial como verdade

O prompt em app/prompts/system_prompt.md restringe a resposta aos trechos recuperados e orienta o modelo a responder em português do Brasil, com linguagem simples.

Contexto

PDF extraído para Markdown

scripts/extract_context.py extrai data/context.pdf, grava data/contexto.md com marcadores de página e gera data/context_index.json.

Indexação

Recriação controlada da coleção

scripts/index_context.py chama RagService.index_context(), que apaga e recria a coleção configurada antes de adicionar os chunks.

Memória

Histórico em memória

MemoryService guarda mensagens por session_id durante a execução do processo e envia as últimas rodadas ao modelo quando aplicável.

Parâmetros

Configurações observadas

LLM_PROVIDER Provedor da LLM: openai ou gemini.
LLM_MODEL Modelo usado para gerar respostas.
CONTEXT_FILE_PATH Caminho do Markdown ou TXT usado como contexto.
SYSTEM_PROMPT_PATH Arquivo com as regras de comportamento da LLM.
VECTORSTORE_PATH Diretório de persistência do ChromaDB.
CHUNK_SIZE e CHUNK_OVERLAP Tamanho dos chunks e sobreposição usada na indexação.
TOP_K Quantidade máxima de trechos retornados pela busca.
EMBEDDING_PROVIDER Define embeddings locais ou via provedor de LLM.
Execução

Ambiente e empacotamento

O projeto usa requirements.txt para dependências Python, Dockerfile com python:3.12-slim e docker-compose.yml expondo a aplicação na porta 8374.

O arquivo .env não deve ser exposto. Use .env.example como referência de nomes de variáveis e preencha as chaves localmente.
